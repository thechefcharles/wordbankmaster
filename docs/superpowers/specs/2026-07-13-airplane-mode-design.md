# ✈️ Airplane Mode — Design

**Date:** 2026-07-13
**Status:** Approved, ready for implementation plan

## Goal

A fully **offline**, **no-money** practice mode. Play WordBank puzzles with zero
network — on a plane, on the subway, or when you're broke and don't want to wait
for tomorrow's Daily. Same core feel as the Daily (points budget → buy letters →
keep what you don't spend), but every point is for pride only. Never touches Cash.

## Why

Two real gaps this closes at once:

1. **The broke dead-end.** A player with no Cash can't buy into Cash Game and has
   already done today's Daily. Today they hit a wall and leave. Airplane Mode gives
   them something to play, unlimited, for free.
2. **No offline game.** The app is 100% server-authoritative today — every move is a
   Supabase RPC, and boot requires a login. There is nothing to play without WiFi.

We deliberately **drop the Cash payout** that an earlier "Free Play" concept had.
That earlier mode was retired in V2 precisely because it paid out Cash and
cannibalized the economy (leftover ledger reasons `freeplay_reward` /
`freeplay_cashout` are the fingerprints). Offline **cannot** safely pay Cash anyway:
an offline client is an untrusted client, and you can never let an untrusted client
mint real money. Points-for-pride sidesteps that entirely. Broke-player recovery
*inside the online economy* is a separate future problem.

## Non-goals (v1)

- **No Cash / no money of any kind.** Points are cosmetic score, full stop.
- **No heat/streak multiplier.** Flat Daily-style loop.
- **No leaderboard and no online points-sync.** Points are device-local.
- **No "bust"/loss.** It's consequence-free practice.
- **No exact rule-parity with the server.** The offline engine only needs to be fun
  and self-consistent; it is not a mirror that must track server changes.

## Architecture

Airplane Mode runs **entirely on the client**, always — whether online or offline the
code path is identical, so there's no "am I connected?" branching inside gameplay.
It leans on facts already true of the app:

- It's a **Capacitor iOS app**, so the web bundle already ships inside the app and the
  shell loads with no network. Only Supabase calls need WiFi — and Airplane Mode makes
  none.
- The **full puzzle catalog is ~58 kB** (1,200 puzzles), small enough to bundle whole.
- **Letter pricing already lives client-side** (`LETTER_COSTS` in `GameStore.js`).

Four new isolated pieces, plus reuse of the existing presentational components:

### Components

**1. `static/airplane-puzzles.json` — bundled puzzle pack**
- A build-time export of `daily_puzzles`: `{ id, phrase, category, clue, subcategory }`.
- ~58 kB, committed as a static asset so it ships in the app bundle.
- Regenerated by a small script (`scripts/export-airplane-puzzles.mjs`) when the pool
  changes; not fetched at runtime.

**2. `src/lib/airplane/engine.js` — the offline game engine (pure, no I/O)**
- Pure functions over a plain state object; no stores, no network, no DOM. Fully
  unit-testable in isolation.
- **State:** `{ puzzleId, phrase, category, clue, revealed:number[], incorrect:string[],
  budget, spent, status:'active'|'solved'|'over' }` (phrase is held in-engine to resolve
  reveals/guesses; the UI only ever sees masked output).
- **`newGame(puzzle)`** → initial state. Computes the **points budget**:
  `budget = round(1.2 × Σ(LETTER_COSTS[c]) for distinct letters c in phrase / 10) × 10`
  (headroom so a skilled solver keeps some; factor is tunable).
- **`buyLetter(state, letter)`** → new state. Charges `LETTER_COSTS[letter]` if
  `budget - spent ≥ cost` and the letter is untried; reveals all its positions if present,
  else appends to `incorrect`. Insufficient budget or a repeat is a no-op (returns state
  unchanged).
- **`submitGuess(state, filled)`** → new state. If every blank matches the phrase →
  `status:'solved'`. Otherwise the guess simply doesn't complete (no penalty, keep
  playing/buying). No bust.
- **`score(state)`** → points earned this puzzle = `budget - spent` **only when solved**,
  else `0`.
- **`maskedBoard(state)`** → the render model (revealed letters + blanks + incorrect set +
  budget/spent), with the answer never included.

**3. `src/lib/airplane/store.js` — device-local persistence + session controller**
- Wraps localStorage (Capacitor Preferences-compatible) under an `airplane:` namespace.
- Persists: **lifetime points**, **best single-puzzle score**, **seen puzzle ids** (for
  no-repeat), **badge milestones reached**, and the **in-progress engine state** (so a
  quit/resume — or app kill mid-flight — restores the current puzzle).
- Exposes a Svelte store for the current game + totals, and actions
  (`start()`, `buy()`, `guess()`, `next()`, `quit()`) that call the engine and persist.
- **Puzzle selection:** random unseen puzzle from the bundle; once all 1,200 are seen,
  reset the seen-set and continue (endless).
- **Badges:** on crossing 10k / 25k / 50k / 100k lifetime points, record the milestone
  and surface a one-off toast. Cosmetic only, device-local.

**4. Entry + offline detection**
- **Play menu:** an **Airplane Mode** card alongside Daily / Cash Game / Challenges.
- **Login screen:** a "Play offline ✈️" affordance so a cold start with no signal still
  reaches a game without logging in.
- **Offline detection:** if the app boots and can't reach the network / has no session,
  surface Airplane Mode as the fallback instead of stranding the user on the loading or
  login screen. (Complements the auth self-heal already in place.)

**5. UI — reuse the presentational components**
- Airplane Mode lives at its own route (`/airplane`) reachable **without a session**, so
  the offline path stays fully decoupled from the auth-gated online game loop in
  `+page.svelte`.
- It reuses the existing **presentational** components — the phrase board and the
  keyboard — driven by the engine's `maskedBoard()` and the store's actions, rather than
  the server-backed `GameStore` daily/climb functions. Exact component reuse vs. light
  re-shell is an implementation detail for the plan; the requirement is that offline
  gameplay never imports the auth/RPC path.

## Data flow

```
bundle JSON ──▶ store.start() ──▶ engine.newGame() ──▶ maskedBoard() ──▶ board + keyboard UI
                     ▲                                                          │
                     └──── store.buy()/guess()/next() ◀── user taps ───────────┘
                                    │
                          engine.buyLetter/submitGuess/score
                                    │
                          persist points/best/seen/progress (localStorage)
```

No Supabase, no `auth.uid()`, no network in the entire loop.

## Error handling & edge cases

- **Corrupt/absent local state** → treat as a fresh start (points 0, no crash).
- **App killed mid-puzzle** (e.g. phone off on a plane) → the persisted in-progress engine
  state restores the exact puzzle on relaunch.
- **All puzzles seen** → reset the seen-set; endless play continues.
- **Bundle missing/malformed** at build time → the export script validates non-empty and
  fails the build rather than shipping an empty pack.
- **Online vs offline** → irrelevant to gameplay; the same client code runs either way.

## Testing

- **Engine unit tests** (pure functions): budget computation, buy reveals correct
  positions, buying an absent letter marks incorrect and charges, insufficient-budget and
  repeat buys are no-ops, guess solves only on a full correct fill, `score` is
  `budget-spent` on solve and `0` otherwise. No DOM/network needed.
- **Store tests:** persistence round-trips (points/best/seen/progress), no-repeat
  selection, seen-set reset at exhaustion, badge milestones fire once.
- **Manual/Playwright:** enter Airplane Mode from the Play menu **with the network
  disabled**, play a puzzle end-to-end, kill+relaunch mid-puzzle to confirm resume,
  confirm zero Supabase requests are made.

## Out of scope (future, explicitly not v1)

- Cosmetic **online leaderboard** + points-sync when reconnected.
- **Heat/streak** depth.
- **Account association** of offline points (device-local only in v1).
- **Broke-player recovery inside the online economy** (a separate design).

## Success criteria

- A user in airplane mode (network fully off), logged in or not, can open the app, reach
  Airplane Mode, and play puzzles indefinitely with points accruing and no errors.
- Zero network requests occur during Airplane Mode play.
- Nothing in Airplane Mode reads or writes Cash, credit, or any server state.
